## Project Overview

Umay is a Mandarin Chinese learning app powered by Google Gemini AI. Originally built for Turkish speakers, it is now multilingual: every AI feature is parameterised by a `(native, target)` language pair (`tr`, `en`, `zh`). UI strings and prompts resolve per-pair via `src/i18n/strings.ts` and `ai_backend/prompts.py::for_pair(native, target)`; missing pairs fall back to `("tr", "zh")`.

## Commands

```bash

npm run dev

npm run dev:frontend       # Express + Vite middleware (port 3000)
```

Run a single Vitest file: `npx vitest run src/lib/__tests__/audioPlayer.test.ts`.

### Python backend (`ai_backend/`)

Managed with `uv` (not pip). Add a dep: `cd ai_backend && uv add <package>`. First-time setup: `uv sync`. Venv lives at `ai_backend/.venv/`. Python 3.13+ required.

## Architecture

```
React SPA (Vite) → Express (:3000) → FastAPI (:8000) → Gemini API
                       │                   └─ google-genai (live) / litellm (REST)
                       └─ static /audio (pre-downloaded WAVs from HF bucket)
```

- **`server.ts`** — Express entry. Rate-limits `/api` at 30 req/min, serves the SPA via Vite middleware (dev) or `./dist` static (prod), serves pre-downloaded audio at `/audio`, and proxies a **specific whitelist** of paths to FastAPI: `/api/tts`, `/api/grammar`, `/api/transcribe`, `/api/evaluate-pronunciation` (HTTP) and `/api/live` (WebSocket upgrade).
- **`ai_backend/main.py`** — FastAPI entry. One router per feature in `ai_backend/routers/`: `tts`, `grammar`, `pronunciation`, `transcribe`, `live`. All are mounted under `/api` prefix. System prompts live in `ai_backend/prompts.py` and are dispatched per language pair.
- **No shared DB.** Vocabulary is a static module (`src/data/vocabulary.ts`) regenerated by `scripts/sync-vocab.ts`. Audio is pre-downloaded files on disk — see caching strategy below.

### Gemini model + library split

| Purpose | Model | Library |
|---|---|---|
| TTS | `gemini/gemini-2.5-flash-preview-tts` | `litellm` (`modalities=["audio"]`, `format: pcm16`) |
| Grammar / transcribe / pronunciation | `gemini/gemini-2.5-flash` | `litellm` |
| Real-time voice dialog | `gemini-2.5-flash-native-audio-dialog` | `google-genai` (`client.aio.live.connect`) |

`live.py` contains an explicit **vLLM swap point** comment — the Gemini live session is intended to be replaceable by a local real-time model implementing the same send/receive contract.

### Audio caching (bucket-first, TTS fallback)

1. `src/data/vocabulary.ts` rows reference URIs of the form `/audio/{field}/{speed}/row_{N}.wav`.
2. `npm run download-audio` pulls WAVs from the HF bucket `Thoria/TTS-UMAY` into `data/audio/{folder}/{speed}/row_{N}.wav`. **URLs are constructed deterministically** (folder × speed × row index); the dataset's JSON audio columns contain NULLs and are NOT authoritative. Idempotent reruns are cheap (skip-if-exists). Intended to run during Docker image build so containers ship with audio.
3. At runtime, `src/lib/audioPlayer.ts` **HEAD-probes the static URL**. 200 → stream the file. 404 → call `/api/tts` for Gemini; the backend returns base64 PCM16 at 24 kHz; the client wraps it with a 44-byte RIFF/WAVE header (`pcm16ToWav`) before handing it to `<audio>`. Nothing is written server-side — there is no `/api/save-audio` endpoint.
4. Turkish target (`tr`) is v2-gated: missing URIs return `null` and the caller hides the play button rather than falling through to TTS (`GEMINI_FALLBACK_TARGETS = ['zh', 'en']`).
5. The Express static mount uses `fallthrough: false` — **critical**. Without it, missing audio files pass through to Vite's SPA middleware, which returns `index.html` with `Content-Type: text/html`, which the browser then fails to decode as audio. The explicit 404 is what lets the client trigger TTS fallback.

### Live Tutor WebSocket

Client → `/api/live` → Express upgrades and proxies to FastAPI → `live.py` opens a Gemini `aio.live.connect` session.

Wire protocol:
- First client message must be `{type: "config", voiceName, systemInstruction, model?}`. Anything else closes the socket with code 1008.
- Subsequent client messages: `{type: "audio", data: base64, mimeType: "audio/pcm;rate=16000"}`.
- Server messages: `{type: "connected" | "message" | "error"}`. Audio is relayed as Gemini's native `serverContent.modelTurn.parts[i].inlineData`. Barge-in surfaces as `serverContent.interrupted: true`.

Client audio pipeline (`src/hooks/useLiveAPI.ts`): mic captured at **16 kHz PCM16**, Gemini responses played back at **24 kHz** via a scheduled `AudioBufferSourceNode` queue (drift-resistant — `nextPlayTime` advances by buffer duration so chunks stay seamless).

## Environment

- `GEMINI_API_KEY` in `.env.local` — **required by both sides**. Frontend reads it via `vite.config.ts` `define` at build time (so it ends up in the bundle — a Docker build-arg in compose). Backend reads it via `ai_backend/config.py`, which raises at import time if unset.
- `AI_BACKEND_URL` — defaults to `http://localhost:8000`. Set to `http://ai:8000` in Docker compose.
- `AI_BACKEND_PORT` — Python side (default 8000). `PORT` — Express side (default 3000).

## Frontend path alias

`@/*` → project root. Declared in `tsconfig.json`, `vite.config.ts`, and `vitest.config.ts`. All three must stay in sync when adding path mappings.

## Testing

- Frontend: Vitest with `jsdom`, `globals: true`. Test files: `src/**/*.test.ts(x)` and `scripts/**/*.test.ts`. `passWithNoTests: true` so empty dirs don't fail CI.
- Python backend has no test suite yet.

## Docker

Two services in `docker-compose.yml`:
- `ai` — Python 3.13 + uv, exposes `:8000`, healthcheck hits `/health`.
- `frontend` — 3-stage Node build, depends on `ai` being healthy. `GEMINI_API_KEY` is passed as a **build-arg** (baked into the bundle).

They share a named volume `audio_files` mounted at `/app/data/audio` so the `download-audio` script's output persists and is reachable by both.

## Gotchas

- Don't hand-edit `src/data/vocabulary.ts` — it's generated. Run `npm run sync-vocab`.
- The rate-limit middleware (`/api` scope) runs **before** the proxy, so hits against non-whitelisted `/api/*` paths still consume quota even though they 404.
- Client-side Gemini API key exposure is intentional for this prototype; if this changes, the `define` in `vite.config.ts` must also change and all frontend callers must route via `/api/*`.
- The `/audio` static mount is separate from the `/api` proxy — audio requests never touch FastAPI.
